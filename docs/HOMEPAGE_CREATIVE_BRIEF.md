# TWOG Autoresearch Homepage Creative Brief

Status: v1 planning artifact  
Last updated: 2026-04-30  
Audience: founder, designer, motion designer, frontend implementer

## Objective

Design a homepage for the complete future-state TWOG Autoresearch platform, not
only the current repository status.

The page should position TWOG Autoresearch as serious research infrastructure
for canine hemangiosarcoma and related human translational oncology. It should
show how evidence enters the system, becomes organized, supports retrieval,
enables specialist research agents, connects to scientific tools, and produces
source-traceable outputs for human review.

The homepage should be ambitious but bounded. It should not imply cures,
diagnosis, prescription, clinical decision-making, or agents proving medical
truth.

## Core Positioning

TWOG Autoresearch is an agent-assisted research platform for canine
hemangiosarcoma and translational oncology.

It is not just a literature database. It is a research operating system:
evidence comes in, gets structured, becomes searchable, agents work on top of
it, scientific tools test ideas, and humans review the outputs.

## Homepage Spine

The page should communicate one progression:

```text
Evidence enters
  -> Evidence is structured
  -> Agents work on it
  -> Scientific tools test assumptions
  -> Humans review source-traceable outputs
```

This progression should drive the copy, visuals, motion, and page rhythm.

## Section Plan

### 1. Hero: The Research Engine

Purpose: establish the complete future-state platform.

Message: TWOG Autoresearch is a living, agent-assisted research platform for
HSA and translational oncology.

Visual: full-viewport animated research engine. Biomedical sources flow into a
structured evidence library, then into agent and scientific-tool lanes.

Motion: slow evidence-flow animation. Source nodes become chunks, entities,
claims, citations, and embeddings. Agent lanes activate from the structured
library. Human review stays visible as the final gate.

### 2. The Problem: Evidence Is Fragmented

Purpose: explain why this system needs to exist.

Copy:

```text
Canine hemangiosarcoma is aggressive, under-served, and biologically complex.
Relevant evidence is scattered across veterinary oncology, human angiosarcoma,
vascular tumor biology, sarcoma literature, drug databases, protein references,
omics datasets, safety records, and emerging online signals. Important
connections are easy to miss.
```

Visual: scattered source fragments across literature, databases, clinical
records, compounds, proteins, omics, and social leads.

Motion: fragments organize into source lanes as the user scrolls.

### 3. The Platform: From Evidence To Research Workflows

Purpose: show the complete system.

Copy:

```text
TWOG Autoresearch continuously ingests research sources, structures them into an
evidence library, prepares them for retrieval, and gives specialist agents
access to the information and tools needed to generate source-traceable research
artifacts.
```

Visual: system map.

Flow:

```text
Ingest -> Structure -> Retrieve -> Agent Work -> Scientific Tools -> Human Review
```

Motion: scroll-driven pipeline activation. Each layer should reveal what it
adds, not just animate for decoration.

### 4. Evidence Ingestion Layer

Purpose: show source breadth without implying uncontrolled scraping.

Sources to represent:

- PubMed
- Europe PMC
- PMC OA
- OpenAlex
- ChEMBL
- PubChem
- UniProt
- RCSB PDB
- OpenFDA animal events
- veterinary and clinical sources
- omics and safety datasets
- X/Twitter research leads
- future partner or lab-submitted data

Copy:

```text
The platform is designed to ingest both traditional scientific sources and
monitored research leads. Social posts and online discussions are treated only
as leads, then routed back to papers, datasets, authors, compounds, or primary
sources before entering the evidence workflow.
```

Visual: source cards entering dedicated ingestion lanes.

Motion: each lane shows status: harvested, triaged, queued, or needs review.

### 5. The Evidence Library

Purpose: explain how raw information becomes usable.

Copy:

```text
Raw records are normalized into research objects, document chunks, entities,
claims, citations, and embeddings. This creates a searchable evidence library
that supports RAG, MCP tools, source review, citation-backed briefs, and future
scientific workflows.
```

Visual: product-like interface mock, not generated art.

Elements to show:

- research objects
- document chunks
- entities
- claims
- citations
- embeddings
- source health
- retrieval-ready status

Motion: rows populate, filters apply, citations stay attached to claims.

### 6. Research Agent Layer

Purpose: introduce agents as bounded research workers.

Copy:

```text
Specialist agents operate on top of the evidence library. They can search across
sources, compare findings, identify candidate mechanisms, design hypotheses,
plan validation work, inspect contradictory evidence, and produce
source-traceable reports for human review.
```

Agent types:

- Evidence Scout Agent
- Translational Hypothesis Agent
- Skeptic Review Agent
- Source Follow-Up Agent
- Full-Text Ops Agent
- Validation Planning Agent
- Scientific Tool Runner Agent

Visual: agent workbench. Agents should look like accountable workflows, not chat
bubbles or generic AI personas.

Motion: task cards move from the evidence library into agent lanes. Each output
must retain visible citations or evidence references.

### 7. Scientific Tool Layer

Purpose: make the frontier future state concrete.

Copy:

```text
As the platform expands, agents will connect to scientific research tools
through Docker and GPU-backed environments such as RunPod. These workflows may
include molecular modeling, protein-compound analysis, structure review, omics
interpretation, safety signal review, pathway comparison, and computational
validation tasks.
```

Visual: scientific tooling console with molecular, protein-compound, omics,
safety, and GPU job lanes.

Possible future tool categories:

- molecular modeling
- protein-compound analysis
- structure comparison
- omics interpretation
- pathway analysis
- safety and adverse-event review
- docking or simulation workflows
- GPU-backed validation tasks

Motion: a hypothesis moves into tool lanes; outputs return as reviewable
artifacts with status, assumptions, and source links.

### 8. Hypothesis Pipeline

Purpose: show how research moves forward without overclaiming.

Copy:

```text
The system is designed to move from evidence to hypotheses in a controlled
sequence: ingest sources, structure evidence, retrieve relevant context,
generate multi-perspective briefs, evaluate synthesis quality, identify
candidate hypotheses, plan validation steps, run appropriate tools, and route
outputs to human experts.
```

Visual: vertical or horizontal pipeline.

Stages:

```text
Evidence -> Brief -> Evaluation -> Hypothesis -> Validation Plan -> Tool Run -> Expert Review
```

Motion: gated progression. Weak evidence routes to review instead of being
promoted.

### 9. Quality And Oversight

Purpose: build credibility.

Copy:

```text
Every output is designed to remain traceable. Research briefs require citations.
Synthesis outputs are evaluated for citation coverage, perspective balance,
contradiction handling, novelty, actionability, and readiness for review. Human
oversight remains central, especially for clinical interpretation, research
prioritization, and validation decisions.
```

Visual: quality gate dashboard.

Metrics to show:

- citation coverage
- perspective balance
- contradiction handling
- novelty
- actionability
- readiness for review
- human review status

Motion: quality checks pass, warn, or route to review. Avoid celebratory
"breakthrough" animation.

### 10. Current Build Status

Purpose: maintain trust by being explicit about what is live versus planned.

Copy:

```text
TWOG Autoresearch is under active development. Core ingestion, source
monitoring, full-text handling, embeddings, research brief generation, synthesis
evaluation, Dagster orchestration, and MCP-style access are being built
incrementally. The platform is not a medical device, diagnostic system, or
treatment recommendation engine.
```

Visual: lane status board.

Status groups:

- Live or active build: ingestion, source monitoring, embeddings, research
  briefs, synthesis evaluation, Dagster orchestration, MCP-style access.
- Expanding: full text depth, source coverage, agent review flows, lead
  follow-up.
- Future: GPU tool execution, molecular modeling, omics workflows, partner
  validation loops.

Motion: minimal. This section should feel operational and factual.

CTA routes:

Primary CTA stack:

- Discuss Collaboration

- Follow Project Updates

## Motion System

Motion should communicate system behavior, not decoration.

Use motion for:

- evidence moving from messy to structured
- citations staying attached to outputs
- agents operating inside bounded lanes
- scientific tools producing reviewable artifacts
- weak evidence routing to review
- source health and quality gates updating

Avoid motion for:

- AI magic effects
- glowing orbs
- fast particle fields
- constant blinking
- fake breakthrough moments
- autonomous decision-making visuals

Recommended implementation:

- Hero: code-native canvas/WebGL/Three.js evidence graph.
- Section reveals: scroll-driven, restrained.
- Product surfaces: real HTML/CSS components, not generated dashboard images.
- Scientific tool visual: mix of product UI plus a real or generated molecular
  visual.

## Visual System

Tone:

- rigorous
- clinical but not sterile
- technical but readable
- serious but not cold
- founder-led, not corporate biotech

Palette:

- near-black or deep graphite base
- off-white text
- evidence blue/cyan for data flow
- signal green for validated states
- amber for uncertainty or review
- controlled red/magenta accent for vascular biology or risk

Avoid:

- one-note dark blue SaaS
- purple AI gradients
- biotech stock-photo gloss
- medical miracle imagery
- purely decorative bokeh/orb backgrounds

Typography:

- strong editorial headline
- clean product UI type
- compact labels for system diagrams
- avoid oversized text inside cards or dashboards

## Asset Checklist

### Build Code-Native First

1. Hero animated evidence graph
   - Sources become structured evidence.
   - Evidence routes to agents and tools.
   - Human review remains visible.

2. Evidence library interface
   - Research objects, chunks, claims, entities, citations, embeddings.
   - Should be designed as future product UI.

3. Agent workbench interface
   - Agents receive tasks and return source-traceable artifacts.
   - Avoid chat-only presentation.

4. Quality gate dashboard
   - Citation coverage, contradiction handling, readiness, human review.

### Generate Only If Needed

1. Subtle biomedical texture
   - Non-stock, abstract vascular/tissue/cell motif.
   - Should sit behind product visuals, not dominate.

2. Molecular/protein-compound visual
   - For scientific tool layer.
   - Can be generated, rendered from real structure data, or built with 3D.

3. Optional short atmospheric video loop
   - 8 to 12 seconds.
   - No baked-in text.
   - Only if the code-native hero feels too sterile.

### Do Not Generate

- Fake unreadable dashboards.
- Images implying treatment success.
- Agents represented as human doctors.
- Videos with clinical claims.
- Visuals that imply the platform diagnoses, prescribes, or proves therapies.

## Current Frontend Implications

The existing homepage direction should be revised away from broad
"AI-driven treatment discovery" language and toward the research-platform
frame. The new page should lead with evidence infrastructure, agent-assisted
research workflows, validation planning, and human-reviewed outputs.

Specific old-positioning patterns to retire:

- claims that imply treatment discovery is already solved
- large simulation or compound counts without source context
- terminal visuals that make the system look more clinically conclusive than it
  is
- copy that says or implies "new cancer treatments" without qualification

The new page can still be ambitious. The ambition should come from the full
research operating system, not from overstated medical outcomes.
